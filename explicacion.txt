1. DISEÑO DE LAS ESTRUCTURAS DE DATOS Y ALINEACIÓN

A. Estructura 'Usuario' (definida en usuarios.h):
   - char usuario[50];    -> Reserva 50 bytes para el string del identificador.
   - char clave[50];      -> Reserva 50 bytes para la contraseña del vendedor.
   - char nombre[50];     -> Reserva 50 bytes para el nombre completo.
   * Justificación del orden: Se agrupan consecutivamente las cadenas de caracteres de tamaño idéntico para facilitar el mapeo directo desde la función 'strtok' al leer 'vendedores.txt'.

B. Estructura 'Producto' (definida en productos.h):
   - char codigo[50];     -> 50 bytes (Código único del producto).
   - char nombre[50];     -> 50 bytes (Nombre comercial).
   - int cantidad;        -> 4 bytes (Entero para inventario disponible).
   - float costo;         -> 4 bytes (Punto flotante para precio de compra).
   - float precio_venta;  -> 4 bytes (Punto flotante para precio de venta).
   * Justificación del orden: Sigue de manera estricta y secuencial la disposición del archivo 'producto.txt'. Colocar primero los arreglos de caracteres ('char') y al final los tipos numéricos primitivos ('int' y 'float') minimiza el desperdicio por alineación de memoria (padding) que introduce el compilador GCC, garantizando un uso eficiente de la memoria RAM.

C. Estructura 'DetalleVenta' (definida en productos.h):
   - Mantiene un orden análogo a 'Producto' omitiendo el inventario global, pero introduciendo 'cantidad_a_vender' (4 bytes). Permite almacenar en un buffer dinámico temporal los ítems de la compra actual antes de ser validados y escritos en 'ventas.txt'.


2. GESTIÓN DE MEMORIA DINÁMICA (HEAP) Y CRECIMIENTO GEOMÉTRICO
Para evitar el desbordamiento de memoria y permitir que el sistema procese un número indeterminado de productos y ventas sin colapsar, se implementó la estrategia de asignación dinámica:
- Inicialización: Al arrancar la lectura de inventarios y el inicio de la orden, se reserva memoria exacta en el Heap utilizando la función 'malloc()'.
- Crecimiento Dinámico: En caso de que el archivo 'producto.txt' o el rellenado de la factura actual superen la capacidad inicialmente estimada (10 para productos y 5 para el detalle de venta), el sistema reasigna espacio duplicando geométricamente su tamaño actual (capacidad = capacidad * 2) mediante la función 'realloc()'. Esto reduce drásticamente la fragmentación de memoria y el costo computacional de reasignar recursos constantemente.
- Liberación: Una vez que el ciclo de facturación se procesa, se ejecuta la función 'free()' sobre los punteros de los arreglos dinámicos para devolver los bloques de memoria al sistema operativo, erradicando por completo los 'memory leaks' (fugas de memoria).


3. LOGICA DEL FLUJO DE DATOS Y VALIDACIONES COMPLEMENTARIAS
- El orden de las operaciones asegura la integridad del negocio: primero se valida al usuario (máximo 3 intentos).
- Al buscar un producto, se realiza un recorrido secuencial comparando cadenas mediante 'strcmp()'.
- Si hay stock suficiente, se resta preventivamente del arreglo en memoria. Si el usuario cancela la transacción al final, la memoria se libera sin alterar el archivo original. Solo cuando se introduce la confirmación ("si"), se vuelcan los datos actualizados a 'producto.txt' y 'ventas.txt', garantizando la persistencia exacta de los datos.
